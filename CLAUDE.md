## What Fullball is

A free-to-play **football-agent** game for iPhone, themed around the 2026 World
Cup. You play a representative ("representante") building a roster of clients:

- **Scout** unknown talent in packs (gacha — cheap, random), or
- **Sign** specific marquee clients on the **transfer market** (pay Cash — certain), then
- **Field** clients in **live matches**; when they perform you earn a **Cash commission**, **Rep**, points, and **Form/Scouts**, which you reinvest in bigger clients.

Core loop: **sign → field → live → earn → reinvest → climb the agency ranks.**

This repo is a **complete, offline, playable MVP vertical slice**. Everything is
mock/local — no backend, no real IAP, no auth.

## Hard constraints (non-negotiable — these shaped every decision)

1. **iPhone only**, **portrait locked**. No iPad / Mac / visionOS, no size-class code.
2. **SwiftUI only** (UIKit only where SwiftUI has no equivalent — currently just `UIImage` for portrait loading).
3. **Swift 6**, strict concurrency (`SWIFT_STRICT_CONCURRENCY: complete`). Deployment target **iOS 17.0** (Observation).
4. **MVVM** with the `@Observable` macro. **Never** `ObservableObject`/`@Published`/`@StateObject`/`@ObservedObject`.
5. **No real player likeness** — no real names, faces, kits, club/federation crests, or sponsor marks. Players are fictional (`ARG #10` tag + a generated funny name). "Famous" = in-universe marquee (Icon tier).
6. **Virtual currency only** — no real-money wagering / cash-out / crypto. Shippable under App Store rule 3.1.1, not 5.3.
7. **Disclose gacha odds** in the UI (Odds sheet on Scout).
8. **Minimal TDD** — tests only for deterministic economy/gacha/generation logic (Swift Testing). No view/navigation/SwiftData-wiring tests.
9. Prefer **zero third-party packages** (SPM + Apple frameworks only). Currently zero deps.
10. Develop against **SwiftUI Previews + mock data**, not full clean builds.

> Workflow prefs: **no "Co-Authored-By" trailer** in commits. Plans/design docs are auto-approved — don't block on approval. Use `context7` MCP to confirm current SwiftUI/SwiftData/Swift-concurrency APIs before implementing.

## Tech stack

- Swift 6.x, SwiftUI, `NavigationStack`, iOS 17+, iPhone portrait.
- State: Observation (`@Observable` VMs owned by `@State`, `@Bindable` for two-way).
- Persistence: **SwiftData** (`@Model`) for owned/mutable data; bundled JSON (`Codable`) for the static catalog.
- Concurrency: `async/await`, `@MainActor` VMs/services, `AsyncStream` for the live feed.
- Tests: **Swift Testing** (`import Testing`, `@Test`, `#expect`).
- Project generated by **XcodeGen** (`project.yml`). The `.xcodeproj` is **gitignored** — regenerate it.

## Build, run, test

```bash
brew install xcodegen          # one-time
xcodegen generate              # regenerate Fullball.xcodeproj (after ANY file add/remove or project.yml change)

# build
xcodebuild build -project Fullball.xcodeproj -scheme Fullball \
  -destination 'platform=iOS Simulator,name=iPhone 15'

# test (41 tests, 8 suites — economy/gacha/generation only)
xcodebuild test -project Fullball.xcodeproj -scheme Fullball \
  -only-testing:FullballTests \
  -destination 'platform=iOS Simulator,name=iPhone 15'
```

CLI install + launch on a booted sim:
```bash
SIM=<udid>                     # e.g. an iPhone 15 (iOS 17)
APP=$(find ~/Library/Developer/Xcode/DerivedData/Fullball-*/Build/Products/Debug-iphonesimulator -maxdepth 1 -name Fullball.app | head -1)
xcrun simctl install $SIM "$APP" && xcrun simctl launch $SIM com.fullball.app
```

### Dev launch args (UserDefaults / `NSArgumentDomain`)
Pass after the bundle id, e.g. `xcrun simctl launch $SIM com.fullball.app -seedDemo 1 -startTab 2`:
- `-seedDemo 1` — on first run, seed a roster + a lineup whose nations are in the live slate (so earners rows populate). Behind a flag; never runs normally.
- `-startTab N` — open a tab (0 Scout · 1 Market · 2 Roster · 3 Live · 4 Agencies).
- `-didSeeIntro YES` — skip the first-run onboarding cover.
- `-avatarStyle` — (removed) no longer used.
- `APIFootballKey` (Info.plist or arg) — when set, the catalog loads from api-football at startup (fictionalized); absent ⇒ bundled JSON (default, offline).

## Conventions

- **Theme**: all colors/fonts via `WC` ([Theme.swift](Fullball/Features/Components/Theme.swift)). Adaptive (light/dark) neutrals are asset colorsets (`ScreenBG/CardBG/InkText/LineColor/FillBG/FillDeep/SubText/FaintText`); brand colors are literals. `WC.spectrum` = WC2026 multicolor. Never hardcode hex in views — add to `WC`.
- **ViewModels**: `@MainActor @Observable final class`, one per screen, built in the View's `init` from the injected `AppContainer` (`_vm = State(initialValue: …)`). No business logic in views.
- **Services**: protocol-first, `@MainActor` where they touch SwiftData. Concrete impls + `Mock…`. ViewModels depend on protocols. Wired in `AppContainer` (the only composition root) and injected via `.environment`.
- **Determinism**: anything "random but stable" is seeded — `SeededRandomProvider` (SplitMix64) + FNV-1a hash of an id/string. Used by gacha (injected RNG), `FixtureGenerator`, `NameGenerator`, `AvatarAssets`, `DeviceSeed`.
- **Portraits**: bundled JPGs in `Resources/Avatars/`, mapped by card id via `AvatarView`/`AvatarAssets` (top-anchored fill for cards; `CardPortraitFull` for the detail screen).
- **Funny names**: `NameGenerator.funnyName(for: id)` → stable fictional name; shown as the card title, with `nationTag #num · POS` as the identity line.

## Gotchas (will bite you)

- **Regenerate after adding/removing files**: `xcodegen generate`. The `.xcodeproj` is gitignored.
- **`.stringsdata` "Multiple commands produce" error**: fixed by `SWIFT_EMIT_LOC_STRINGS: NO` in `project.yml`. Don't flip it back.
- **Adding a SwiftData model**: also add it to `AppContainer.schema` or it won't persist (and may crash on relaunch with a mismatched store).
- **Ambiguous `-destination`**: pin a specific sim (`name=iPhone 15` can match multiple OS versions; use a udid for screenshots).
- **Sim screenshots**: `xcrun simctl io <udid> screenshot`. Driving taps needs computer-use, which requires macOS Screen-Recording permission for the Claude app (often ungranted) — so navigation-dependent screens may be unverifiable from here; build + logic still hold.
- **Strict concurrency**: static `NSCache`/shared mutable state must be `@MainActor` or `Sendable`. `@Model` types aren't `Sendable` — keep them main-actor.
- **`@Sendable` AsyncStream closures**: don't capture `var`s — bind to a `let` first (see `MockLiveMatchService`/`LiveMatchService`).

## What Fullball is

A free-to-play **football-agent** game for iPhone, themed around the 2026 World
Cup. You play a representative ("representante") building a roster of clients:

- **Scout** unknown talent in packs (gacha — cheap, random), or
- **Sign** specific marquee clients on the **transfer market** (pay Cash — certain), then
- **Field** clients in **live matches**; when they perform you earn a **Cash commission**, **Rep**, points, and **Form/Scouts**, which you reinvest in bigger clients.

Core loop: **sign → field → live → earn → reinvest → climb the agency ranks.**

This repo is a **complete, playable MVP vertical slice**. The core loop is
offline/local; a **Firebase backend** (anonymous-first auth + cloud save +
shared leaderboard + shared match slate) was added on `main` — see the Firebase
section in [ROADMAP](docs/ROADMAP.md). Still stubbed: real IAP (StoreKit), push,
localization.

## Hard constraints (non-negotiable — these shaped every decision)

1. **iPhone only**, **portrait locked**. No iPad / Mac / visionOS, no size-class code.
2. **SwiftUI only** (UIKit only where SwiftUI has no equivalent — currently just `UIImage` for portrait loading).
3. **Swift 6**, strict concurrency (`SWIFT_STRICT_CONCURRENCY: complete`). Deployment target **iOS 17.0** (Observation).
4. **MVVM** with the `@Observable` macro. **Never** `ObservableObject`/`@Published`/`@StateObject`/`@ObservedObject`.
5. **No real player likeness** — no real names, faces, kits, club/federation crests, or sponsor marks. Players are fictional (`ARG #10` tag + a generated funny name). "Famous" = in-universe marquee (Icon tier).
6. **Virtual currency only** — no real-money wagering / cash-out / crypto. Shippable under App Store rule 3.1.1, not 5.3.
7. **Disclose gacha odds** in the UI (Odds sheet on Scout).
8. **Minimal TDD** — tests only for deterministic economy/gacha/generation logic (Swift Testing). No view/navigation/SwiftData-wiring tests.
9. Prefer **zero third-party packages** (SPM + Apple frameworks only). **One deliberate exception:** the **Firebase iOS SDK** (FirebaseAuth + FirebaseFirestore + FirebaseStorage + FirebaseAppCheck), added for the backend — quarantined behind service protocols in `Services/Auth` + `Services/Firestore`; never import Firebase in Views/ViewModels (App Check is the one bootstrap-only exception, in `FullballApp`).
10. Develop against **SwiftUI Previews + mock data**, not full clean builds.

> Workflow prefs: **no "Co-Authored-By" trailer** in commits. Plans/design docs are auto-approved — don't block on approval. Use `context7` MCP to confirm current SwiftUI/SwiftData/Swift-concurrency APIs before implementing.

## Tech stack

- Swift 6.x, SwiftUI, `NavigationStack`, iOS 17+, iPhone portrait.
- State: Observation (`@Observable` VMs owned by `@State`, `@Bindable` for two-way).
- Persistence: **SwiftData** (`@Model`) for owned/mutable data; bundled JSON (`Codable`) for the static catalog. **Backend:** Firestore is the durable source of truth (server-authoritative); SwiftData is the local synchronous cache, kept in sync by write-through decorators.
- Concurrency: `async/await`, `@MainActor` VMs/services, `AsyncStream` for the live feed.
- Tests: **Swift Testing** (`import Testing`, `@Test`, `#expect`).
- Project generated by **XcodeGen** (`project.yml`). The `.xcodeproj` is **gitignored** — regenerate it.
- Backend: **Firebase** (Auth + Firestore + Storage) via SPM; `GoogleService-Info.plist` is **gitignored** (injected in CI, see below).

## Build, run, test

```bash
brew install xcodegen          # one-time
xcodegen generate              # regenerate Fullball.xcodeproj (after ANY file add/remove or project.yml change)

# build  (iPhone 15 sim is gone on current Xcode — use iPhone 16; pin a udid to avoid ambiguity)
xcodebuild build -project Fullball.xcodeproj -scheme Fullball \
  -destination 'platform=iOS Simulator,name=iPhone 16'

# test (108 tests — economy/gacha/generation + futsal engine + energy + cloud DTO mapping + nonce; no view/wiring tests)
xcodebuild test -project Fullball.xcodeproj -scheme Fullball \
  -only-testing:FullballTests \
  -destination 'platform=iOS Simulator,name=iPhone 16'
```

> **Firebase build note:** the app needs `Fullball/GoogleService-Info.plist` (gitignored) at runtime; `FirebaseApp.configure()` is guarded behind its presence. Without it the app *builds* fine but traps on the first Firebase call (e.g. anonymous sign-in). Linking an Apple ID needs an iCloud-signed device/sim.

CLI install + launch on a booted sim:
```bash
SIM=<udid>                     # e.g. an iPhone 15 (iOS 17)
APP=$(find ~/Library/Developer/Xcode/DerivedData/Fullball-*/Build/Products/Debug-iphonesimulator -maxdepth 1 -name "*.app" | head -1)
xcrun simctl install $SIM "$APP" && xcrun simctl launch $SIM com.juanledesma.Fulbo.app
```

### Dev launch args (UserDefaults / `NSArgumentDomain`)
Pass after the bundle id, e.g. `xcrun simctl launch $SIM com.juanledesma.Fulbo.app -seedDemo 1 -startTab 2`:
- `-seedDemo 1` — on first run, seed a roster + a lineup whose nations are in the live slate (so earners rows populate). Behind a flag; never runs normally.
- `-startTab N` — open a tab (0 Scout · 1 Market · 2 Roster · 3 Live · 4 Agencies).
- `-didSeeIntro YES` — skip the first-run onboarding cover.
- `-avatarStyle` — (removed) no longer used.
- `APIFootballKey` (Info.plist or arg) — when set, the catalog loads from api-football at startup (fictionalized); absent ⇒ bundled JSON (default, offline).

### Xcode Cloud / CI
`ci_scripts/ci_post_clone.sh` runs on clone and (in order):
1. **Injects `GoogleService-Info.plist`** by base64-decoding the Xcode Cloud secret env var `GOOGLE_SERVICE_INFO_PLIST_BASE64` into `Fullball/` (the plist is gitignored — the repo is public). Without this the archived app traps on first Firebase call.
2. Runs `xcodegen generate` (installs xcodegen via Homebrew if missing). The `.xcodeproj` is gitignored.
3. **Copies a tracked `ci_scripts/Package.resolved`** into the generated project's `…/xcshareddata/swiftpm/`. Xcode Cloud disables automatic SwiftPM resolution (an explicit `-resolvePackageDependencies` also fails there, exit 74), so the pinned resolved file must be supplied. **Keep `ci_scripts/Package.resolved` in sync** when bumping Firebase/SPM deps (regenerate locally, copy it back).

The scheme `Fullball` is shared (in `xcshareddata`) so Xcode Cloud finds it post-generation. Bundle id `com.juanledesma.Fulbo.app` matches the ASC record (display/store name "Fullball Manager"). (Same env-var pattern works for the api-football key.)

## Conventions

- **Theme**: all colors/fonts via `WC` ([Theme.swift](Fullball/App/Theme.swift)). Adaptive (light/dark) neutrals are asset colorsets (`ScreenBG/CardBG/InkText/LineColor/FillBG/FillDeep/SubText/FaintText`); brand colors are literals. `WC.spectrum` = WC2026 multicolor. Never hardcode hex in views — add to `WC`. (Note: token accessors are short, e.g. `WC.sub` not `WC.subText`, `WC.screenBG`, `WC.inkText`, `WC.coral`, `WC.display(_:)`, `WC.ui(_:)`.)
- **ViewModels**: `@MainActor @Observable final class`, one per screen, built in the View's `init` from the injected `AppContainer` (`_vm = State(initialValue: …)`). No business logic in views.
- **Services**: protocol-first, `@MainActor` where they touch SwiftData. Concrete impls + `Mock…`. ViewModels depend on protocols. Wired in `AppContainer` (the only composition root) and injected via `.environment`.
- **Determinism**: anything "random but stable" is seeded — `SeededRandomProvider` (SplitMix64) + FNV-1a hash of an id/string. Used by gacha (injected RNG), `FixtureGenerator`, `NameGenerator`, `DeviceSeed`.
- **Portraits**: fetched from Firebase Storage `players/{id}.jpg` by `PlayerImageStore` (memory NSCache → disk cache → Storage download); injected via `\.playerImageStore`. `AvatarView`/`CardPortraitFull` are async with a rarity-tinted position-symbol placeholder while loading/offline. Old `avatar_NNN.jpg` files in `Resources/Avatars/` are unused (removable).
- **Player names**: authored per-card (mononym for regulars; mononym + " — epithet" for icons), stored as `name`/`epithet` fields on `Player`. `NameGenerator` is a fallback only (used when a card has no authored name).

## Gotchas (will bite you)

- **Regenerate after adding/removing files**: `xcodegen generate`. The `.xcodeproj` is gitignored.
- **`.stringsdata` "Multiple commands produce" error**: fixed by `SWIFT_EMIT_LOC_STRINGS: NO` in `project.yml`. Don't flip it back.
- **Adding a SwiftData model**: also add it to `AppContainer.schema` or it won't persist (and may crash on relaunch with a mismatched store).
- **Ambiguous `-destination`**: pin a specific sim (`name=iPhone 15` can match multiple OS versions; use a udid for screenshots).
- **Sim screenshots**: `xcrun simctl io <udid> screenshot`. Driving taps needs computer-use, which requires macOS Screen-Recording permission for the Claude app (often ungranted) — so navigation-dependent screens may be unverifiable from here; build + logic still hold.
- **Strict concurrency**: static `NSCache`/shared mutable state must be `@MainActor` or `Sendable`. `@Model` types aren't `Sendable` — keep them main-actor.
- **`@Sendable` AsyncStream closures**: don't capture `var`s — bind to a `let` first (see `MockLiveMatchService`/`LiveMatchService`).

## Current state (1-paragraph)

Full agent loop: Scout (gacha w/ pity + 50/50 + disclosed odds), Market (transfer signings), Roster (collection, filters, DEX %, squad rating, portrait tiles with energy bar + ×N copies badge + sort control, card detail with full portrait + train/limit-break + Gem energy refill), Live (active-only futsal tactics match — see below), Agencies (leaderboard with Firestore-backed agency rename), Wallet bar (Cash/Gems/Scouts/Rep), daily drop, Rep→pull exchange, first-run intro. **Firebase backend on `main`**: anonymous-first auth, cloud-saved wallet/collection/pity + live progress, shared real leaderboard, shared match slate. **Futsal tactics match** (branch `feat/futsal-tactics-match`): tap PLAY → Match Setup (positional 5-a-side field GK/DEF/MID/MID/FWD; drag players from roster strip into slots; off-position = 0.5× stats; an **Auto-fill** button picks the strongest squad via `SquadAutoFill`) → pick Intensity (Conservative/Balanced/Aggressive) + Focus (Defend/Balanced/Attack); a live Monte-Carlo win-odds bar (`FutsalOdds`, above the pitch) updates as you slot players + change tactics (draws fold as ½ so an even contest reads ~50%) → watch deterministic futsal sim on a horizontal pitch (card-portrait players, animated ball, live scoreline + clock, event feed) → full-time rewards. `FutsalEngine` (pure, tested) runs 14 possessions; midfield strength is normalized to a full outfield so filling the team is strictly additive; the best finisher takes each shot; player-style RPS (`PlayStyle`: pace>physical>technical) adds a shooting edge. AI opponents via `OpponentGenerator` (stats ×`FutsalRules.opponentStrengthMultiplier` for difficulty). Authored catalog stats are compressed at load by `StatSquish` (client-side stopgap — see ROADMAP to bake at source). Energy system: every `CardInstance` has `energy` (0–100), drains per match (base 20, ×Intensity, captain +10), regens ~4/hr, Gem refill on Card Detail. Old persistent Lineup XI, `LineupService`, and `Lineup` SwiftData model removed. Cards show position (GK/DEF/MID/FWD) not shirt number. All currency-spend buttons use the new `CurrencyCost` component. 108 tests pass. **Stubs / next**: StoreKit Gem IAP (monetization = StoreKit-only), api-football loader (opt-in), real PvP opponents, push/localization. See [ROADMAP](docs/ROADMAP.md) for the prioritized next steps.
